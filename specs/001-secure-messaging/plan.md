# Implementation Plan: Hybrid Post-Quantum Secure Messaging

**Branch**: `001-secure-messaging` | **Date**: 2026-02-25 | **Spec**: [spec.md](spec.md)
**Input**: Feature specification from `/specs/001-secure-messaging/spec.md`

## Summary

Build a secure messaging application using .NET 10 and the Blazor Web App hybrid template that implements the Signal Protocol with post-quantum extensions. All cryptographic operations (X3DH key exchange with ML-KEM-768, Double Ratchet, AES-256-GCM encryption) execute exclusively in the Blazor WebAssembly client using BouncyCastle.Cryptography. The ASP.NET Core server acts as an untrusted relay — storing only encrypted ciphertext, public keys, and metadata. Real-time delivery uses SignalR; persistence uses SQL Server with EF Core Code First. All entity primary keys use `decimal(28,8)` generated by `Toledo.SharedKernel.Helpers.DecimalTools.GetNewId()`.

## Technical Context

**Language/Version**: C# / .NET 10 (LTS)
**Primary Dependencies**: BouncyCastle.Cryptography 2.6.2, ASP.NET Core Identity, SignalR, EF Core 10
**Storage**: SQL Server 2022 (server-side via EF Core Code First) + Browser IndexedDB (client-side)
**Testing**: xUnit v3 3.2.2, BenchmarkDotNet 0.15.8
**Target Platform**: Web (Blazor WebAssembly + ASP.NET Core server); future: .NET MAUI mobile
**Project Type**: Web application (Blazor Web App hybrid template)
**Performance Goals**: Key exchange <500ms, message encrypt/decrypt <50ms, <1KB hybrid overhead, <2s delivery
**Constraints**: All crypto in WASM (pure managed C# only), zero plaintext on server, 10K concurrent users
**Scale/Scope**: 10K concurrent users, up to 10 devices per user, 100 group participants
**Primary Key Strategy**: All entity IDs use `decimal(28,8)` via `Toledo.SharedKernel.Helpers.DecimalTools.GetNewId()` — no Guids

**Structure Decision**: Blazor Web App hybrid template with 5 source projects:
1. `Toledo.SharedKernel` — Shared utilities including `DecimalTools` for ID generation. Referenced by all other projects.
2. `ToledoMessage` — ASP.NET Core server host (API + SignalR + EF Core).
3. `ToledoMessage.Client` — Blazor WASM client (all crypto + interactive UI).
4. `ToledoMessage.Shared` — DTOs, enums, and constants shared between server and client.
5. `ToledoMessage.Crypto` — Cryptographic library (BouncyCastle wrappers). Referenced only by Client.

The Crypto project is referenced only by `ToledoMessage.Client` (WASM) — the server never references it, enforcing the zero-trust boundary at the project dependency level.

**Project dependency graph**:
```
Toledo.SharedKernel ← ToledoMessage.Shared ← ToledoMessage
                                            ← ToledoMessage.Client ← ToledoMessage.Crypto
```

## Complexity Tracking

> **Justified deviations from Constitution:**

| Violation | Why Needed | Simpler Alternative Rejected Because |
|-----------|------------|-------------------------------------|
| BouncyCastle for classical crypto (Principle III says libsodium/System.Security) | Blazor WASM cannot load native libraries (libsodium). BouncyCastle is pure managed C# and works everywhere including WASM. | Using two libraries (BouncyCastle for PQ + native for classical) is impossible in WASM; single-library approach is actually simpler. |
| .NET 10 instead of .NET 8 (Principle V specifies .NET 8) | User explicitly requested .NET 10. Constitution says "or latest LTS" — .NET 10 is the next LTS (Nov 2026). | .NET 8 is valid but user made a deliberate choice for the latest runtime. |
| Blazor Web App hybrid instead of pure Blazor WebAssembly (Principle V) | User explicitly requested the Blazor "Web App" template (hybrid). Hybrid provides faster initial loads via SSR while crypto components use InteractiveWebAssembly. | Pure WASM has slower initial load (full runtime download before any render). Hybrid is objectively better UX. All crypto still runs in WASM. |
| Separate Toledo.SharedKernel project (5 src projects instead of 3) | User requires `DecimalTools.GetNewId()` in namespace `Toledo.SharedKernel.Helpers`. This is a cross-cutting utility needed by Server, Client, and Shared. | Putting it in ToledoMessage.Shared would work but breaks the user's explicit namespace convention. |
| Separate ToledoMessage.Crypto project (adds another project) | Enforces zero-trust boundary at the dependency level — server project cannot accidentally reference crypto internals. Crypto tests can run independently. | Putting crypto code in the .Client project works but loses architectural enforcement. The crypto module is complex enough (~15 files) to justify separation. |
| decimal(28,8) IDs instead of Guid | User explicitly required `DecimalTools.GetNewId()` as the only way to get primary keys. Time-ordered IDs improve SQL Server clustered index performance. | Guid is simpler but user made a deliberate architectural choice. |

## Phased Delivery Strategy

### Phase 1: Foundation (Setup + Infrastructure)
- Solution structure, project scaffolding, NuGet packages
- `Toledo.SharedKernel` project with `DecimalTools.cs`
- EF Core DbContext, entity models with `decimal(28,8)` PKs, initial migration
- ASP.NET Core Identity (registration, login, JWT)
- Basic Blazor layout and routing

### Phase 2: Cryptographic Library (ToledoMessage.Crypto)
- Classical primitives: X25519, Ed25519, AES-256-GCM wrappers
- Post-quantum primitives: ML-KEM-768, ML-DSA-65 wrappers
- Hybrid operations: combined KEM, combined signatures, HKDF
- Protocol: X3DH with PQ extension, Double Ratchet
- Key management: identity generation, pre-key generation, fingerprints
- Full test suite for all crypto operations

### Phase 3: User Story 1 — Account Registration & Key Generation (P1)
- Registration API endpoint + client UI
- Client-side key generation (identity + pre-keys)
- Pre-key bundle upload to server
- Security fingerprint display

### Phase 4: User Story 2 — Initiating a Secure Conversation (P1)
- User search endpoint + UI
- Pre-key bundle fetch
- X3DH session establishment (client-side)
- First encrypted message send/receive
- Conversation creation API

### Phase 5: User Story 3 — Real-Time Message Exchange (P1)
- SignalR hub + client connection
- Double Ratchet integration for per-message keys
- Real-time message delivery
- Delivery and read status indicators
- Offline message queuing and retrieval
- Message ordering via sequence numbers

### Phase 6: User Story 4 — Security Verification (P2)
- Safety number computation and display
- Conversation verification marking
- Key change warning notifications

### Phase 7: User Story 5 — Multi-Device Support (P2)
- Device registration and management
- Fan-out encryption (encrypt per recipient device)
- Device linking/unlinking UI
- Pre-key bundle per device

### Phase 8: User Story 6 — Group Messaging (P3)
- Group conversation creation
- Group key distribution via pairwise channels
- Key rotation on membership changes
- Group admin management

### Phase 9: User Story 7 — Disappearing Messages (P3)
- Timer configuration per conversation
- Client-side expiry enforcement
- Server-side message cleanup (for delivered messages)

### Phase 10: Polish & Hardening
- Rate limiting middleware
- Performance benchmarks and optimization
- Security hardening review
- IndexedDB encryption at rest
- Pre-key replenishment automation
- Error handling and edge cases
