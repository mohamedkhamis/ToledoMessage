## Path Conventions

This is a .NET solution with this structure:
- **Solution file**: `ToledoVault.slnx` (at repo root)
- **Server project**: `src/ToledoVault/` (ASP.NET Core host)
- **Admin client project**: `src/ToledoVault.Admin/` (NEW — Blazor WASM)
- **Shared project**: `src/ToledoVault.Shared/` (DTOs, constants)
- **Admin test project**: `tests/ToledoVault.Admin.Tests/` (NEW — MSTest)
- **Build props**: `src/Directory.Build.props` (auto-versioning)

## Existing Codebase Patterns (MUST follow these)

**Entity models**: Classes in `src/ToledoVault/Models/` with public properties. IDs are `long` type generated by `Toledo.SharedKernel.IdGenerator.GetNewId()`. Example: `public long Id { get; set; }`.

**EF Core configurations**: One file per entity in `src/ToledoVault/Data/Configurations/` implementing `IEntityTypeConfiguration<T>`. Use `.HasKey(e => e.Id)`, `.Property(e => e.Id).ValueGeneratedNever()` for long IDs. Use `.HasIndex()` for indexes. Use `.HasData()` for seed data.

**DbContext**: `src/ToledoVault/Data/ApplicationDbContext.cs` — add `public DbSet<EntityName> EntityNames { get; set; }` for each new entity.

**Controllers**: Inherit from `BaseApiController` (provides `GetUserId()`, `TryGetUserId()`). Use `[Route("api/admin/[controller]")]` for admin controllers. Use `[Authorize(Roles = "admin")]` for auth. Return `IActionResult`.

**Services**: Register in `src/ToledoVault/Program.cs` via `builder.Services.AddScoped<ServiceName>()` or `AddSingleton<>()`.

**DTOs**: Sealed records in `src/ToledoVault.Shared/DTOs/`. Use `sealed record ClassName(Type Param1, Type Param2);` syntax.

**JSON serialization**: `long` IDs serialize as strings via `LongToStringConverter` to avoid JS precision loss.

**Password hashing**: Use `Microsoft.AspNetCore.Identity.PasswordHasher<T>` — same as existing `AuthController`.

**JWT tokens**: HS256 signing with `Jwt:SecretKey` from config. Claims: `sub`, `unique_name`, `role`, `jti`. Validate with `TokenValidationParameters`.

**Test pattern**: MSTest with `[TestClass]` and `[TestMethod]`. Use EF Core `InMemoryDatabase` for DB tests. Use `NullLogger<T>.Instance` for logger dependencies.

## Dependencies & Execution Order

### Phase Dependencies

- **Setup (Phase 1)**: No dependencies — can start immediately
- **Foundational (Phase 2)**: Depends on Setup completion — BLOCKS all user stories
- **User Story 1 (Phase 3)**: Depends on Foundational (Phase 2) — this is the MVP
- **User Story 2 (Phase 4)**: Depends on Foundational (Phase 2) — can run in parallel with US1 (different controllers/pages)
- **User Story 3 (Phase 5)**: Depends on Foundational (Phase 2) — can run in parallel with US1/US2 (different controllers/pages)
- **User Story 4 (Phase 6)**: Depends on US1 completion (needs settings infrastructure)
- **Polish (Phase 7)**: Depends on all user stories being complete

### User Story Dependencies

- **User Story 1 (P1)**: Depends only on Phase 2 (Foundational). This is the MVP — admin auth, settings CRUD, and settings UI.
- **User Story 2 (P2)**: Depends only on Phase 2 (Foundational). Independent from US1 — different DB table (LogEntries), different controller, different page.
- **User Story 3 (P3)**: Depends only on Phase 2 (Foundational). Independent from US1/US2 — different DB table (LocalizationOverrides), different controller, different page.
- **User Story 4 (P4)**: Depends on US1 (needs GlobalSetting entity and settings service to verify extensibility).

### Within Each User Story

1. DTOs first (shared project, no dependencies)
2. Server services (depend on DTOs and DB entities from Phase 2)
3. Server controllers (depend on services)
4. Client services (depend on DTOs)
5. Client pages (depend on client services and layout from US1)
6. Tests (can run in parallel with implementation, but must validate against actual code)

### Parallel Opportunities

**Phase 1 (Setup)**: T001, T002 can run in parallel (different projects). T003-T005 sequential.

**Phase 2 (Foundational)**: T007-T010 (DTOs) can all run in parallel. T011-T016 (entities + configs) can all run in parallel. T017 (DbContext) depends on entities. T018 (migration) depends on DbContext. T019-T020 (services + Program.cs) can run in parallel after migration.

**Phase 3-5 (User Stories)**: US1, US2, US3 can run in parallel after Phase 2 (different files entirely). Within each story, DTOs are parallel, then services → controllers → client pages are sequential.
